KPI Tracker/ Settings

Settings

App-level configuration. These choices decide how an issue's effective due date is computed, which in turn anchors any KPI target date that is set relative to the due date (e.g. "reached 3 months after the issue is due").

Due Date Rollup

per hierarchy level

Which dates are prioritized between a parent and its children when resolving an issue's due date? Set a rule for each level of your discovered hierarchy. Levels are read at runtime from the site's issue-type configuration — nothing is hard-coded.

i How each rule resolves a due date (bottom-up, memoized per render):
• From children — always use the merged dates of the children (a parent finishes when its last child finishes); ignore the parent's own date.
• From children, then parent — use children if any exist, otherwise fall back to the issue's own date.
• From parent, then children — the issue's own date wins per-field; children fill only the gaps.
• From parent only — never roll up; use the issue's own date.
• Earliest → latest — the widest range across the parent and all children.

Relative KPI target dates

preview

When a KPI target date on an issue is set as an offset instead of a fixed calendar date, the offset is applied to the issue's effective due date computed with the rules above. Changing a child's date can move a parent's rolled-up due date — and therefore the KPI target date — automatically.

i If an anchor can't be resolved yet (no own date and no dated descendants), the resolved target date shows as pending rather than computing a wrong date.
